Changes from original diagram

  • users table REMOVED — system uses staff_roster as the sole user table; all Lambdas (WCUserslist, WCCreateUser, WCUpdateUser, WCLogin) read/write staff_roster
  • submission_answers table REMOVED — individual question answers are not stored; only the AI analysis JSON is saved in survey_submissions.ai_analysis
  • survey_assignment table KEPT — table exists in DB; not yet used by Angular app (future: control which staff cover which survey+ward)
  • staff_roster.name → first_name + last_name FIXED — corrects the FistName typo and normalises split names (matches WCCreateUser / WCUpdateUser Lambda contracts)
  • survey_submissions.ai_analysis JSON schema UPDATED — now includes sentiment (from Engagement tone chip) and trust_signal (from Institutional trust chip) fields
  • survey_submissions.reviewed_by FK FIXED — now correctly references staff_roster.id instead of the removed users table

status ENUM — survey_submissions auto-flagged by submit logic 
-- Possible values
'draft'     -- saved but not submitted
'submitted' -- normal submission
'reviewed'  -- admin has reviewed
'flagged'   -- URGENT: auto-set when:

-- NARRATIVE surveys:
--   Engagement tone chip = "Urgent"

-- ENVIRONMENTAL scans:
--   Safety score    = Low | Critical
--   Env burden      = High | Critical


-- Low Trust Signal derived when:
-- NARRATIVE: trust_signal = "Low" | "Erosion"
-- ENV SCAN:  Safety cls   = "critical"
Future Scope — Tables defined but not yet implemented
survey_assignment — Table exists in DB. Not yet read by the Angular app — all active surveys are currently visible to all field staff. Implementing it would enforce ward coverage (e.g. Briana covers Ward 1 narratives, Kenechi covers Ward 4 scans) and enable gap reporting.

submission_answers — Table defined but not yet used. Survey form answers are currently summarised only via the AI analysis JSON in survey_submissions.ai_analysis. Implementing this would enable per-question analytics, raw data export, and audit trails.
